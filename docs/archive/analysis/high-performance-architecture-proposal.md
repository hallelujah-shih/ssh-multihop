# SSH-Multihop 高性能演进架构方案 (Multiplexing)

## 1. 核心思想：从“单一隧道”到“连接集群”
该方案旨在通过连接复用（Multiplexing）彻底改变原有的 $O(N)$ 连接模式，转为多隧道共享单一物理 SSH 链路，极大提升系统性能与稳定性。

## 2. 关键组件：`ConnectionManager` (连接池)
引入全局连接池层，充当逻辑转发与物理连接之间的抽象桥梁。
*   **特征哈希**：通过 `user@host:port + [hop_chain]` 生成唯一的路径标识。
*   **物理层抽象**：池化层持有 `*ssh.Client`。若路径一致，后续转发任务不再新建连接，直接开启 SSH Channel。

## 3. 分层连接管理模型
### 3.1 生命周期：引用计数与延迟回收 (Lingering)
*   **引用计数**：每个物理连接记录当前的活跃订阅数。
*   **延迟回收**：当订阅数归零时，连接进入闲置态（Idle），保持 60 秒活跃。若期间无新请求再物理关闭。这有效防止了批量更新隧道时的“连接抖动”。

### 3.2 健康检查：探测与订阅分离
*   **Master Health Check**：池化层统筹物理探测，为每条链路维护单一的高频 Keepalive。
*   **Failure Propagation**：物理层探测失败后，通过根 Context 级联杀掉该路径下所有的逻辑 Forward 任务，触发自动重建。

## 4. SSH Agent 与认证扩展
*   **环境变量继承**：优先尝试继承系统 `SSH_AUTH_SOCK`。
*   **外挂密钥解锁**：支持通过受保护的 Unix Socket 向守护进程安全传递加密密钥的 Passphrase。

## 5. 预期收益
*   **建立耗时**：后续隧道的建立从“秒级”降为“毫秒级”。
*   **资源占用**：大幅降低物理 TCP 连接数、内存开销以及跳板机上的并发会话压力。
*   **稳定性**：共享单一心跳降低流量噪声，失效传播机制确保所有隧道状态高度同步。
