# SSH-Multihop 架构深度审查报告 (2026-03-17)

## 1. 概述
本报告对 `ssh-multihop` 的核心架构进行了深度 Review。除了基础的并发与性能问题外，本次 Review 重点引入了 **“连接复用 (Multiplexing)”** 机制，旨在解决资源冗余、连接建立慢以及服务端压力大等核心痛点。

---

## 2. 核心改进点分析

### 2.1 并发安全：解决 ForwardService 的“重复启动”竞态
*   **现状**：`sync()` 循环异步启动隧道。若 SSH 握手慢，下一次循环会重复触发同一隧道的启动。
*   **改进方案**：引入 `pendingForwards` 集合，确保 ID 在启动中时，后续循环跳过该项。

### 2.2 连接复用：从 $O(N)$ 到 $O(1)$ 的物理连接演进
*   **现状**：每个转发任务都建立独立的 SSH 跳板链。10 个经过相同跳板的转发会产生 10 条 TCP/SSH 连接。
*   **改进方案**：引入 `ConnectionManager` (连接池)。
    *   **特征匹配**：基于跳板链配置（用户、主机、端口、密钥）生成唯一 Key。
    *   **按需复用**：请求相同路径时，直接在现有 `*ssh.Client` 上开启新信道 (Channel)。
    *   **收益**：秒级建立后续隧道、极低的服务端负载、显著降低内存与带宽占用。

### 2.3 分层健康检查：解决复用下的探测冲突
*   **现状**：每个 Forward 独立发送心跳，复用下会导致探测包冗余。
*   **改进方案**：
    *   **物理层探测**：由 `ConnectionManager` 为每个物理连接维护单一的高频心跳（如 15s/次）。
    *   **逻辑层订阅**：Forward 实例订阅物理连接的 Context。物理层断开时，通过 Context 取消自动触发所有关联 Forward 的失效与重连。

### 2.4 连接生命周期：引用计数与延迟回收 (Lingering)
*   **现状**：连接随 Forward 的生命周期强绑定。
*   **改进方案**：
    *   **引用计数**：记录有多少个 Forward 正在共享该物理连接。
    *   **空闲延迟关闭**：当引用计数归零时，启动 60 秒（可配置）的定时器。若期间无新需求，再物理关闭。这能有效防止批量重启隧道时的“连接抖动”。

### 2.5 网络性能优化：`SyscallConn` 与 `bidirectionalCopy`
*   **改进方案**：
    *   使用 `SyscallConn` 设置 Socket 选项，避免 `tcpConn.File()` 导致的 FD 冗余与非阻塞模式失效。
    *   `bidirectionalCopy` 只要有一端结束，立即显式关闭双方连接，防止半关闭状态下的死锁。

### 2.6 安全加固：非交互式身份验证
*   **改进方案**：**后台守护进程严禁 `stdin` 交互**。移除 `fmt.Scanln`，加密密钥必须通过外部 Agent 或预加载机制处理。连接构建层遇到认证失败应立即报错并触发 Backoff。

### 2.7 数据库与 API 效能
*   **改进方案**：
    *   `sync()` 采用 `JOIN` 查询一次性加载所有状态，消除 $O(N)$ 查询压力。
    *   API 增加 `sync=true` 选项，支持在响应前进行初次连接尝试。

---

## 3. 演进路线建议图
1.  **Phase 1 (稳定性)**：修复 `pendingForwards` 竞态与 `bidirectionalCopy` 的挂起问题。
2.  **Phase 2 (性能核心)**：实现 `ConnectionManager` 连接池，引入引用计数与分层健康检查。
3.  **Phase 3 (鲁棒性)**：重构认证逻辑为纯非交互式，优化数据库批量查询。

---

## 4. 结论
通过引入 **连接复用**，`ssh-multihop` 将具备承载大规模、复杂拓扑隧道群的能力。分层的健康检查与引用计数机制确保了系统在高效复用的同时，依然保持高度的确定性和资源安全性。
